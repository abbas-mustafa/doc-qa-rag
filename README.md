## ✨ Overview

DocQA is a full-stack **Retrieval-Augmented Generation (RAG)** application. Each user gets private **workspaces**; drop documents into one and ask natural-language questions. Answers are generated by **Google Gemini** and grounded strictly in the uploaded documents — every response cites the source file (and page, where available). Unlike a naive "extract text and embed" pipeline, DocQA understands **tables**, describes **figures and charts** with vision, and **OCRs scanned pages**, so visual context is searchable too.

> **Try it:** **[doc-qa-rag.vercel.app](https://doc-qa-rag.vercel.app)** — sign up with any email, create a workspace, upload a document, and ask away.

---

## 🚀 Key Features

- **🔐 Multi-user & private** — email auth via Supabase; each account's workspaces, documents, and chat history are isolated. Sessions persist and restore where you left off.
- **📚 Intelligent, per-file-type parsing**
  - **PDF** (PyMuPDF): real page numbers, tables extracted as markdown, embedded **figures described by Gemini vision**, and **OCR** for scanned/image-only pages
  - **DOCX** (python-docx): heading structure preserved, tables, embedded images
  - **TXT**: plain text
- **🧩 Dynamic, structure-aware chunking** — chunk size adapts to document shape (dense prose vs. list-like), tables/figures stay intact, splits respect heading → paragraph → sentence boundaries, with overlap carried for context continuity.
- **🎯 Cited, grounded answers** — semantic retrieval (pgvector cosine similarity, per-workspace), token-optimized prompts (capped context, deduped sources), and inline `[source: file, p.N]` citations.
- **📦 Large uploads** — up to 100 MB, streamed to disk (no memory blow-up).
- **🛡️ Resilient** — automatic retry/backoff on LLM rate limits; graceful failure states.
- **🎨 Polished UI** — animated gradient background, glassmorphism chat, drag-and-drop upload with live processing status, auto-growing input, fully responsive.

## 🏗️ Architecture

```
┌──────────────────────────┐        HTTPS + JWT        ┌──────────────────────────────┐
│  Next.js frontend        │  ───────────────────────▶ │  FastAPI backend (async)     │
│  (Vercel)                │                           │  (Render)                    │
│  • Login / workspaces    │ ◀───────────────────────  │  • Supabase JWT verification │
│  • Chat UI + upload      │      cited answers        │  • parse → chunk → embed     │
└───────────┬──────────────┘                           │  • retrieve → generate       │
            │                                          └───────┬─────────────┬────────┘
            │ Supabase Auth (login/session)                    │             │
            ▼                                                   ▼             ▼
    ┌───────────────┐                              ┌──────────────────┐  ┌──────────┐
    │  Supabase     │                              │ Postgres +       │  │  Google  │
    │  Auth (ES256) │                              │ pgvector         │  │  Gemini  │
    └───────────────┘                              │ (Supabase)       │  │  (LLM +  │
                                                   └──────────────────┘  │  vision) │
                                                                         └──────────┘
```

**Data flow**

1. **Ingest** — Upload → per-file-type parse (text + tables + figures + OCR) → dynamic chunk → embed (Gemini, 768-dim) → store vectors in Postgres. Runs as a background task; the UI polls status.
2. **Ask** — Question → embed → exact cosine retrieval over the workspace's chunks → prompt Gemini with capped, cited context → stream back the answer and persist it.

## 🖥️ Running It Locally

### What you get when you clone

Cloning gives you **all source code** plus the **dependency manifests** (`requirements.txt`, `package.json`) and **env templates** (`.env.example`) — everything needed to *reconstruct* the app. It does **not** include (by design):

- installed dependencies (`.venv/`, `node_modules/`) → you rebuild them
- secrets / `.env` files → you supply your **own** Supabase project + Gemini key
- a database → you point at your **own** Supabase (or any Postgres with `pgvector`) and run the migration

So a fresh clone runs against **your** infrastructure, not the author's. Here's the full path:

### Prerequisites

- **Python 3.12+** and **Node.js 18+**
- A **Supabase** project (free) — gives you both Postgres + pgvector *and* Auth
- A **Google Gemini API key** (free) — [aistudio.google.com/apikey](https://aistudio.google.com/apikey)

### 1. Clone

```bash
git clone https://github.com/abbas-mustafa/doc-qa-rag.git
cd doc-qa-rag
```

### 2. Backend

```bash
cd backend-py
python -m venv .venv
.venv\Scripts\activate            # Windows  ·  source .venv/bin/activate on macOS/Linux
pip install -r requirements.txt

cp .env.example .env              # then edit .env (see table below)
python -m app.migrate             # creates tables + enables pgvector in your DB
python run.py                     # API on http://localhost:8000  (docs at /docs)
```

### 3. Frontend

```bash
cd ../frontend
npm install
cp .env.example .env.local        # then edit .env.local
npm run dev                       # app on http://localhost:3000
```

Open **http://localhost:3000**. With Supabase keys set you'll get the login screen; leave them blank (and set `AUTH_ENABLED=false` in the backend) to run in a no-login local dev mode.

## 🔑 Environment Variables

**`backend-py/.env`**

| Variable | Description |
|---|---|
| `PORT` | API port (default `8000`) |
| `CORS_ORIGINS` | Comma-separated allowed origins (e.g. `http://localhost:3000,https://your-app.vercel.app`) |
| `DATABASE_URL` | Postgres connection string (Supabase **pooler / IPv4** host) |
| `GEMINI_API_KEY` | Google Gemini API key |
| `EMBEDDING_MODEL` | default `gemini-embedding-001` |
| `CHAT_MODEL` / `VISION_MODEL` | default `gemini-2.5-flash` |
| `MAX_FILE_SIZE_MB` | upload limit (default `100`) |
| `AUTH_ENABLED` | `true` to enforce Supabase JWTs; `false` for local dev-bypass |
| `SUPABASE_PROJECT_URL` | your Supabase URL (used to verify JWTs via JWKS) |
| `SUPABASE_JWT_SECRET` | only if your project uses the legacy HS256 secret |

**`frontend/.env.local`**

| Variable | Description |
|---|---|
| `NEXT_PUBLIC_API_URL` | backend base URL |
| `NEXT_PUBLIC_SUPABASE_URL` | Supabase project URL |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Supabase anon / publishable key |

## ☁️ Deployment

The live app runs **frontend on Vercel** and **backend on Render** (both free tier).

- **Frontend → Vercel**: import the repo, set **Root Directory** to `frontend`, add the three `NEXT_PUBLIC_*` env vars, deploy.
- **Backend → Render**: the included [`render.yaml`](render.yaml) is a Blueprint — *New → Blueprint*, pick the repo, then enter the secret env vars (`DATABASE_URL`, `GEMINI_API_KEY`, `CORS_ORIGINS`) in the dashboard.
- **Wire them**: set the backend's `CORS_ORIGINS` to your Vercel URL, and point the frontend's `NEXT_PUBLIC_API_URL` at the Render URL.

> Vercel can't host the backend — it's a long-running server doing background processing and file uploads, which don't fit a serverless model.

### Good to know (free tiers)

- **Render free** services sleep after ~15 min idle; the first request then takes ~50s to wake.
- **Gemini free tier** has strict rate limits — large or figure-heavy documents can exceed the embedding rate. The backend retries with backoff; a paid key is recommended for heavy use.
